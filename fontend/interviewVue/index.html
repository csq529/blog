1.vue生命周期

beforeCreate 在实例初始化new Vue之后，数据观测 (data observer) 和 event/watcher 事件配置之前被调用 ‌created‌：在实例创建完成后被立即调用，此时已完成数据观测、属性和方法的运算、watch/event 事件回调。然而，挂载阶段之前的 render 函数不还不应该被调用。网络请求 beforeMount‌：在挂载开始之前被调用：相关的 render 函数首次被调用。该钩子在第一次服务器端渲染期间不会被调用。 ‌mounted‌：el 被新创建的 vm.$el 替换，并挂载到实例上去之后调用该钩子。如果根实例挂载了一个文档内元素，当 mounted 被调用时 vm.$el 也在文档内 beforeUpdate‌：数据更新时调用，发生在虚拟 DOM 打补丁之前。这里适合在更新之前访问现有的 DOM，比如手动移除已添加的事件监听器。 ‌updated‌：由于数据更改导致的虚拟 DOM 重新渲染和打补丁，在这之后会调用该钩子。当这个钩子被调用时，组件 DOM 已经更新，所以你现在可以执行依赖于 DOM 的操作。然而在大多数情况下你应该避免在此期间更改状态，因为这可能会导致更新循环。 ‌beforeDestroy‌：在实例销毁之前调用。在这一步，实例仍然完全可用。清除定时器 ‌destroyed‌：Vue 实例销毁后调用。调用后，Vue 实例指示的所有东西都会解绑定，所有的事件监听器会被移除，所有的子实例也会被销毁。

2.vue2响应式原理，与vue3有何区别

vue2响应式原理：vue2的核心是object.defineProperty,通过数据劫持 + 订阅发布模式来实现 基本流程： 1.在组件data选项定义数据时，Vue会递归遍历所有属性 2.使用Object.defineProperty()为每个属性添加getter和setter 3.依赖收集：每个组件实例都有一个watcher，当渲染读取数据触发getter时，将watcher添加到该属性的依赖列表中 4.派发更新：当数据修改触发setter时，通知依赖列表中的所有watcher重新渲染 三个主要问题： 无法检测对象属性的添加和删除，必须用Vue.set/Vue.delete 对数组的变动需要特殊处理，Vue 2通过拦截数组的push、pop等7个方法来实现响应式 初始化时需要递归遍历所有属性，深层嵌套对象性能开销大 vue3响应式原理： Vue 3 基于Proxy重写了整个响应式系统，彻底解决了上述问题。 核心实现： 使用new Proxy(target, handler)代理整个对象 handler中定义get、set、deleteProperty等捕获器 在get中做依赖收集（track），在set和deleteProperty中做触发更新（trigger）

两者区别： ‌响应式系统‌ Vue 2‌：使用 Object.defineProperty() 实现响应式，无法监听对象属性的新增/删除，对数组方法需特殊处理。 ‌ Vue 3‌：采用 ‌Proxy‌ 代理整个对象，能监听属性的增删、数组索引变化等，支持更复杂的数据结构，性能更优 ‌ API 风格‌ ‌ Vue 2‌：基于 ‌Options API‌，逻辑按 data、methods、computed、watch 等选项分散组织，大型组件中逻辑碎片化严重。 ‌ Vue 3‌：引入 ‌Composition API‌（通过 setup() 函数），允许按业务逻辑聚合代码，提升可读性与复用性，同时完全兼容 Options API ‌ 模板语法‌ Vue 2‌：组件必须有‌单个根节点‌。 ‌ Vue 3‌：支持 ‌Fragment（多个根节点）‌，无需额外包裹元素 ‌ 生命周期钩子‌ Vue 2‌：使用 beforeCreate、created、beforeMount、mounted 等传统钩子。 ‌ Vue 3‌： beforeCreate 和 created 被移除（逻辑可写在 setup() 中）； 新增组合式生命周期钩子如 onMounted()、onUnmounted() 等； beforeDestroy → beforeUnmount，destroyed → unmounted ‌ ‌ 性能优化‌ ‌Vue 3‌： 虚拟 DOM 算法优化，减少不必要的遍历； 支持 ‌Tree Shaking‌，未使用的 API 不打包进最终产物； 启动更快、内存占用更低 ‌ ‌ 新特性组件‌ ‌Teleport‌：将内容渲染到 DOM 树外任意位置（如模态框）‌ ‌Suspense‌：优雅处理异步组件加载状态 ‌ ‌ TypeScript 支持‌ ‌Vue 3‌：核心代码用 TypeScript 重写，提供更完善的类型推断与 IDE 支持

6.vue2和vue3的diff算法

对比维度 Vue 2.x (双端 Diff) Vue 3.x (快速 Diff + 编译优化)

核心策略 双端比较：通过新旧子节点列表的头尾交叉对比 快速Diff + 最长递增子序列：先处理前缀和后缀的公共节点， 对中间无序部分使用最长递增子序列算法来最小化DOM移动次数。 在新旧列表首尾移动指针来查找可复用节点。

编译优化 全量 Diff：每次更新都会递归遍历所有子节点，无法跳过静态节点。 静态提升 + PatchFlag：编译时标记动态节点（PatchFlag），更新时只对比带标记的

动态内容，直接跳过静态节点，大幅减少比对范围。

列表对比 双端指针移动：通过两端向中间移动的指针，在每一轮中进行最多4次节点匹配尝试。 基于Key的索引映射 + 最长递增子序列：建立新节点在旧列表中的位置 索引数组，通过计算最长递增子序列来确定最少需要移动的节点，实现最高效的移动。

节点移动逻辑 移动已确定：当匹配到节点需要移动时（如旧头=新尾），立即执行DOM移动操作。 先处理再移动：先标记所有可复用节点在新列表中的位置，最后通过算法计算出最小移动集再批量执行。

算法复杂度 理论 O(n)，但在复杂列表乱序场景下性能下降。 理论 O(n)，通过编译优化和算法改进，在绝大多数场景下实际性能远优于 Vue 2。

Vue 2 的 Diff：像一个勤劳但有些“死板”的工人，会仔细检查每一项，并在过程中不断调整。

Vue 3 的 Diff：像一个聪明的策略家，通过编译时标记来跳过无关工作，在列表对比时能计算出最优方案，用最少的步骤完成任务。

7.vue组件通信

props/Emit->父子组件通信->父通过props向子传递数据，子通过$emit向父发送事件
$refs->父组件获取子组件实例->父组件通过ref获取子组件实例，直接调用其方法或访问数据
$parent/$children->父子组件通信->父组件通过$children获取子组件实例，直接调用其方法或访问数据(vue3已移除)
provide/inject->跨级组件通信->父组件通过provide向子组件提供数据，子组件通过inject获取数据
EventBus->任意组件通信->通过一个独立的 Vue 实例作为事件总线，实现 $on / $emit
vuex/pinia->任意组件（复杂应用）->全局状态管理，所有组件共享同一份响应式状态

Vuex / Pinia（全局状态管理）
State：全局响应式数据
Getter：派生状态（类似 computed）
Mutation：同步修改 state（Vuex 独有）
Action：异步操作，最终提交 mutation
Module：模块化拆分

8.Vuex和Pinia的区别

Mutation：Vuex必须通过 mutation 同步修改 state，pinia则不需要,直接修改state
Action: Vuex必须通过 action 异步修改 state，pinia则不需要,直接修改state
模块化:：Vuex 通过模块化拆分，pinia 每个 store 自动独立，无需嵌套
TypeScript 支持:Vuex较弱，需要手动声明类型,pinia 原生极佳，自动类型推断
体积:gzip Vuex~20KB（gzip 后约 7KB）, pinia~2KB（gzip 后约 1KB）

Pinia 是 Vuex 的“去繁就简”版：移除了 mutation，原生支持 TypeScript，API 更接近 Composition API，体积更小，是 Vue 官方推荐的下一代状态管理方案。

9.Vue的路由实现, hash路由和history路由实现原理说一下

Hash 模式通过监听 hashchange 事件实现，不需要服务端支持；History 模式通过 pushState + popstate 实现，URL 更美观，但需要服务端配置 fallback 到 index.html 来解决刷新 404 问题。

10.说一下 v-if 与 v-show 的区别

如果初始条件大概率是 false，用 v-if 可以避免不必要的渲染。频繁切换的场景，v-if每次都销毁/重建（开销大）,v-show只改 CSS（开销小）推荐使用 v-show

v-if 是"真正的条件渲染"，条件为假时元素不存在于 DOM 中，切换开销大，适合运行时不常变化的场景；v-show 只是 CSS 的 display 切换，元素始终存在，切换开销小，适合频繁切换显隐的场景。

11.v-if和v-for的优先级

不要在同一元素上同时使用 v-if 和 v-for。Vue 2 中 v-for 优先级更高会导致性能浪费；Vue 3 中 v-if 优先级更高但无法访问循环变量。正确的做法是使用计算属性提前过滤数据。

12.keep-alive的常用属性有哪些及实现原理

keep-alive可以实现组件缓存，当组件切换时不会对当前组件进行卸载。
常用的两个属性include/exclude，允许组件有条件的进行缓存。
两个生命周期activated/deactivated，用来得知当前组件是否处于活跃状态。

13.nextTick 的作用是什么？他的实现原理是什么？

nextTick 是 Vue 提供的延迟执行方法，它的核心作用是：在下次 DOM 更新循环结束之后执行回调函数。简单说：等待 Vue 完成 DOM 更新后，再执行你的代码。
nextTick 的核心原理是利用 JavaScript 的事件循环机制，将回调函数推入微任务队列或宏任务队列，在当前同步代码执行完后执行。

nextTick 利用 JavaScript 事件循环机制，将回调延迟到下次 DOM 更新循环之后执行。Vue 内部优先使用 Promise（微任务），依次降级为 MutationObserver、setImmediate 和 setTimeout，确保能在 DOM 更新完成后获取最新状态。

14.说一下 Vue SSR 的实现原理

定义：SSR（Server-Side Rendering，服务端渲染） 是指将 Vue 组件在服务端渲染成 HTML 字符串，直接返回给浏览器，而不是在客户端（浏览器）中动态生成 DOM。

Vue SSR 的核心原理是在服务端调用 renderToString 将 Vue 组件渲染为 HTML 字符串返回给浏览器，然后在客户端执行激活（hydration）复用 DOM 并绑定事件，实现首屏快速渲染和 SEO 优化。它要求同一套代码能在 Node.js 和浏览器中同构运行，因此需要特殊处理数据预取、状态注水和环境判断。 优点：更好的 SEO、更快的首屏加载、对低端设备友好 缺点：服务器压力大、开发复杂度高、调试困难、部署复杂

15.Vue 组件的 data 为什么必须是函数

核心原因：保证每个组件实例拥有独立的数据副本 Vue 组件的 data 必须是函数，是为了保证每个组件实例拥有独立的数据副本。因为组件可能被多次复用，若 data 是对象，所有实例将共享同一份数据，导致一个实例修改数据影响其他实例。而根实例只会创建一次，因此允许 data 为对象。

16.说一下 Vue 的 computed 的实现原理

computed 的本质是一个特殊的 Watcher（Vue 2）或 effect（Vue 3），通过 dirty 标记实现懒计算和结果缓存。当依赖的响应式数据变化时，只将 dirty 设为 true，下次访问时才重新计算，从而避免不必要的重复执行，提升性能。

17.computed 和 watch 的区别

computed 用于派生数据（通过已有数据计算或转换得到的新数据），有缓存，必须同步返回一个值，适合模板中需要的数据处理；watch 用于监听数据变化并执行副作用，支持异步，适合数据变化后需要执行的操作（如请求 API、操作 DOM、持久化存储）。两者不可互相替代，应根据场景选择。

18.说一下 Vue complier 的实现原理是什么样的？

Vue 编译器通过三个阶段（解析 → 优化 → 代码生成）将模板字符串转换为渲染函数。解析阶段生成 AST，优化阶段标记静态节点，代码生成阶段输出可执行的渲染函数字符串。生产环境推荐使用预编译（不含编译器版本），以减少运行时开销。

19.Vue 与 Angular 以及 React 的区别是什么？

简单来说，Vue 旨在提供渐进式的灵活性，React 追求极致的声明式UI和生态自由，而 Angular 则是一个面面俱到的“企业级全家桶”。它们的核心区别，决定了它们在不同场景下的适用性。 没有绝对的“最好”，只有相对的“最适合”。Vue是“小而美”的渐进式选择，React是“灵活性至上”的生态之王，而Angular是“大而全”的企业级标配。了解你的项目需求、团队情况，才能做出明智的决策。

22.vue 中的 spa 应用如何优化首屏加载速度?

通过路由懒加载、CDN加速外部库、开启Gzip压缩、按需引入UI库及图片懒加载，可显著减少首屏资源体积与加载时间。

23.Vue 中的 Key 的作用是什么？

key 的核心价值是给 Vue 一个提示，告诉它每个节点的唯一身份，从而让虚拟 DOM 的 diff 算法更高效、更准确，同时避免意外的状态复用问题。

24.组件中写 name 选项有哪些好处

在 Vue 组件中定义 name 选项，主要便于递归调用自身、在 DevTools 调试时清晰识别以及配合 keep-alive 精准缓存。

25.说一下 ref 的作用是什么？

获取 DOM 元素或组件引用，进行命令式操作

26.你的接口请求一般放在哪个生命周期中？为什么要这样做？

放在组件挂载完成后的生命周期中（useEffect/mounted），原因是不阻塞首次渲染、避免服务端执行、且能安全操作 DOM 或进行清理。 created 钩子函数中调用异步请求，因为在 created 钩子函数中调用异步请求有以下优点： 能更快获取到服务端数据，减少页面 loading 时间 SSR 不支持 beforeMount 、mounted 钩子函数，所以放在 created 中有助于代码的一致性 created 是在模板渲染成 html 前调用，即通常初始化某些属性值，然后再渲染成视图。如果在 mounted 钩子函数中请求数据可能导致页面闪屏问题
